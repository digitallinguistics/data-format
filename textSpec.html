The DLX Data Format for a Text

Version É‘.10 (October 2, 2015)

Introduction

This document is a data format specification or schema that lays out a standard format for representing a linguistic text in JSON. (For a brief overview of JSON and how it can be used to represent linguistic data, check out our DLX Data Format page.) Any application designed to use DLX-formatted data must therefore support processing of data that is correctly formatted according to this specification.

How to read this document

  • When a term is represented using code, it refers to a JSON object and will begin with a capital letter. When a term is represented using normal text, it refers to the linguistic object and will begin with a lowercase letter. For example, Word refers to the JSON object that stores data about a word, while word refers to the linguistic unit.
  • Below is a list of attributes (i.e., properties, fields) that a Text object may contain. Other attributes are possible and allowed, but this specification does not require that applications support any attributes beyond the ones listed here. Attributes must be coded using the field name provided in this specification, e.g. using the form "lang", as specified in this document, and not the full word "language".
  • The title of each section of this document is the English name for an attribute, and the way in which this attribute is referred to in this and other documentation.
  • Each attribute is one of four types:
    • an object, represented by curly braces
    • an array, represented by square brackets
    • a text string, represented by double quotes
    • a number, represented by a number sign
  • In some cases, an attribute may be of more than one type. For example, the "translation" attribute may contain either a single string , such as "translation": "The End.", or an object containing multiple strings, such as "translation": { "en": "The End.", "sp": "El Fin." }. In cases like this, both options are listed under separate subsections of the same attribute.
  • Attributes marked as required must be included in their specified object, even if they contain no content. For example, every text must have a "title" attribute, but the value of that attribute may be an empty string "".
  • Attributes marked as optional may be omitted if there is no data present, however applications must still be able to support this field in the cases where the data is present.
  • The description is a brief prose explanation of this particular attribute, along with any notes or special considerations regarding implementation.
  • The field name is way that this attribute should be represented in the actual JSON data. The field name is always provided in code with double quotes, e.g., "token".
  • If the data in a text field must be in a particular format or structured in a certain way, that format will be specified as a regular expression in the field named Regular expression that attributes must match.
  • If each of the items in a collection must be unique, the unique items property will be listed as true.
  • An example is provided for each attribute. In order to make the examples more visually understandable and save space, occasionally a JSON object will be given an abbreviated label, e.g. {Phrase}.
